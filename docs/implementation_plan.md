# AI量化交易系统实施计划

## 目标描述
基于提供的项目规格说明书构建一个全面的AI量化交易系统。该系统集成了用于数据的AKShare、用于策略引擎的FinRL、基于LLM的智能决策LangChain、用于回测的Backtrader/Pandas、用于后端的FastAPI以及用于前端Web应用的Refine。系统架构遵循6层设计，具备Docker容器化、全面的监控以及事件驱动/异步任务处理能力。

## 需要用户审核
> [!IMPORTANT]
> 规格说明书描述了一个庞大的、生产级别的分布式系统。为了确保我们高效地推进，请审核以下建议的实施方案：
> 
> **实施策略：** 鉴于系统的复杂性，我建议采用分阶段推出的方式，而不是试图一次性构建完整的6层架构。
> - **第一阶段：** 重点建立核心的**基础设施、数据层和后端API**。我们将搭建Docker Compose环境（PostgreSQL/TimescaleDB、Redis、Celery），并使用AKShare实现自动化的数据获取管道。
> - **第二阶段：** 构建**智能决策层和策略引擎**，确保LangChain + FinRL的集成能够与收集到的数据顺畅协作。
> - **第三阶段：** 使用Refine开发**应用层（前端）**，以可视化数据、回测结果并与AI组件进行交互。
> - **第四阶段：** 扩展至高级监控（Prometheus/Grafana）和生产就绪状态。
>
> 请确认这种分阶段的方案是否符合您的期望，或者您是否希望从其他起点开始（例如，在容器化之前先构建一个最小化的单体MVP）。

## 未决问题
> [!WARNING]
> 1. **数据范围：** 对于初始的数据收集管道，您是否有特定的股票代码列表（例如沪深300）和历史时间范围（例如过去5年）作为首批目标？
> 2. **LLM访问权限：** 您是否已经准备好集成智能策略代码生成所需的API密钥（OpenAI或Anthropic）？
> 3. **回测引擎：** 规格说明中同时提到了“Backtrader”和“利用pandas的向量化模式”。虽然Backtrader非常适合事件驱动的回测，但向量化的pandas对于简单策略来说速度更快。我建议我们在初始的MVP回测引擎中先从向量化pandas开始以追求速度，然后再将Backtrader纳入用于复杂的、事件驱动的回测。您同意吗？

## 拟议的变更

我们最初将重点放在 **第一阶段：基础设施和数据层设置**。

### 1. 基础设施层 (Docker及基础设置)
- 初始化 `docker-compose.yml` 文件以配置：
  - **PostgreSQL / TimescaleDB**：用于关系型和时序数据存储。
  - **Redis**：用于缓存和Celery消息代理。
- 创建数据库架构的初始初始化脚本（日线/实时K线数据表）。

### 2. 后端基础 (FastAPI & Celery)
- 设置Python项目结构（使用Poetry或pip）。
- 创建FastAPI应用程序骨架，配置基本的CORS和Swagger UI。
- 配置SQLAlchemy以连接到TimescaleDB实例。
- 配置Celery以处理后台任务（例如，每日AKShare数据获取）。

### 3. 数据层 (AKShare集成)
- 为A股股票数据创建数据模型（代码、开盘价、收盘价、最高价、最低价、成交量等）。
- 实现AKShare数据获取脚本。
- 实现数据清洗和摄取例程（处理异常、插入数据库）。
- 创建基本的REST API以查询存储的股票数据。

## 验证计划

### 自动化测试
- 运行 `docker-compose up` 以验证所有基础设施服务启动时没有错误。
- 为AKShare数据获取函数编写单元测试，以确保数据格式正确。

### 手动验证
- 访问FastAPI Swagger UI以确保API正在运行。
- 触发手动数据获取接口，并使用数据库客户端验证数据是否出现在PostgreSQL/TimescaleDB数据库中。
- 通过检查临时任务状态是否被缓存来测试Redis连接。
